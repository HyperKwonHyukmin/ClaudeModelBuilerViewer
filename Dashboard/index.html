기하 Primitive — 왜 직접 구현했는가?
설계 이유

외부 수학 라이브러리(MathNet 등)를 쓰지 않고 직접 구현한 이유는, Cmb.Core가 외부 NuGet 의존성 없이 순수 BCL만으로 동작해야 하기 때문입니다.

BCL = .NET 기본 클래스 라이브러리. 별도 설치 없이 .NET에 기본 포함되어 있는 것.

타입특성주요 멤버 및 설명
Point3 readonly struct, IEquatable<Point3> X, Y, Z (double) · DistanceTo() · 연산자 +/− (Vector3와 교차 연산) · readonly struct = 스택 할당, 복사 시 참조 문제 없음
Vector3 readonly struct Normalize() — 길이를 1로 만든 단위벡터 · Dot(내적) — 두 벡터 평행도 측정, 투영 계산 · Cross(외적) — 두 벡터에 수직인 벡터, 보 방향(Orientation) 계산에 사용
Segment3 readonly struct ClosestPointTo(Point3 p) — 선분 위에서 p에 가장 가까운 점 반환 · t 파라미터를 [0,1]로 clamp하여 선분 끝점 밖으로 나가지 않도록 보장 · NodeEquivalence, Intersection Stage에서 핵심 사용

A-2 도메인 모델 + JSON 직렬화 완료

FEM 엔티티 — 왜 sealed class/record를 섞어 썼는가?
설계 이유

Node, BeamElement 등 런타임에 많이 생성·수정되는 객체는 record로 설계했습니다.

값 동등성(두 객체의 내용이 같으면 동일하다고 판단)이 필요한 경우 record의 자동 생성 Equals가 편리합니다.

타입종류주요 멤버 및 설명
NodeTags [Flags] enum 한 노드가 여러 특성을 동시에 가질 수 있음 — None=0, Weld=1, Intersection=2, Merged=4, Boundary=8 — | 연산으로 조합: NodeTags.Weld | NodeTags.Intersection
BeamSectionKind enum H | L(앵글) | Rod(원형봉) | Tube | Box | Channel(채널) | Bar(평철) — BDF PBEAML 카드의 TYPE 필드에 직접 대응
EntityCategory enum Structure | Pipe | Equipment — BDF 출력 시 단면 종류 결정, U-bolt snap 대상 판별(Structure만 대상)에 사용
Node record int Id, Point3 Position, NodeTags Tags
BeamElement record Id, StartNodeId, EndNodeId, PropertyId, EntityCategory, Vector3 Orientation
RigidElement record Id, IndependentNodeId(마스터), IReadOnlyList<int> DependentNodeIds(슬레이브) — Nastran RBE2 카드에 대응
PointMass record Id, NodeId, double Mass — U-bolt 또는 Equipment 집중 질량
BeamSection record Id, BeamSectionKind Kind, double[] Dims(치수 배열), int MaterialId
Material record Id, Name, E(탄성계수), Nu(포아송비), Rho(밀도) — 기본 Steel: E=206000, Nu=0.3, Rho=7.85e-9
IdAllocator class int Next() — 단조 증가 ID 발급, 절대 재사용 없음. FeModel 내 모든 엔티티가 공유 사용
FeModel 컨테이너 (Cmb.Core/Model/Context/)
역할

모든 FEM 엔티티의 최상위 컨테이너. LengthUnit = "mm" 고정 (Nastran 입력 단위).

보유 컬렉션: List<Node> / List<BeamElement> / List<RigidElement> / List<PointMass> / List<BeamSection> / List<Material>

JSON 직렬화 — Source Generator 방식을 선택한 이유
구현 방식

런타임 리플렉션 없이 컴파일 타임에 직렬화 코드 생성 → AOT(Native) 배포 가능.

리플렉션 = 런타임에 타입 정보를 읽는 방식. AOT = 실행 전에 미리 컴파일하는 방식으로, 리플렉션을 쓸 수 없음.

camelCase 필드명으로 PRD 6.1 JSON 스키마와 1:1 매핑.

API: FeModel.ToJson() / FeModel.FromJson(string json) 확장 메서드로 호출 간편화.

C-2 전처리 3단계 Stage 구현 대기

Stage 1 SanityPreprocessStage — 전처리 검증
목적
이후 알고리즘이 오작동하지 않도록 "잘못된" 요소를 미리 제거합니다. 중복/단축/공선 3가지를 처리합니다.
알고리즘
중복 요소 제거: 완전히 동일한 시작·끝점을 가진 요소 삭제 (선형 스캔).
단축 요소 제거: 길이 < ShortElemMin(5mm) 인 요소 삭제. 너무 짧은 요소는 수치 불안정 유발.
공선 노드 정리: 3개 노드 A-B-C에서 B가 AC 선분 위에 있으면 중간 노드 B 제거.
Tolerance
ShortElemMin = 5.0 mm

DUPLICATE_ELEM SHORT_ELEM COLLINEAR_NODE

Stage 2 MeshingStage — 요소 분할
목적
너무 긴 빔을 일정 길이 이하로 잘게 쪼갭니다. FEM 해석 정밀도 확보를 위해 필요합니다. Equipment는 분할하지 않습니다.
알고리즘
분할 개수 = ceil(Length / MaxLength) 로 계산.
IdAllocator로 중간 노드를 새로 발급 후 원래 요소를 교체.
분할된 조각들은 원래 요소와 동일한 단면/재료 유지.
Tolerance
MeshingMaxLengthStructure = 2000.0 mm
MeshingMaxLengthPipe = 1000.0 mm

Stage 3 NodeEquivalenceStage — 노드 동일성 판별
목적
아주 가까이 있는 노드들을 하나로 합칩니다. CSV 파싱 시 발생하는 좌표 오차를 제거해 위상을 정규화합니다.
알고리즘 — sweep-and-prune
1) X축으로 노드 정렬 → O(N log N).
2) 슬라이딩 윈도우: X 좌표 차이 > NodeMergeTol(1mm)이면 윈도우 이동.
3) 윈도우 내 노드 쌍끼리 3D 거리 계산 → 1mm 이내면 UnionFind로 묶기.
4) 각 그룹의 최소 ID 노드 1개만 남기고 나머지 삭제.
5) BeamElement / RigidElement의 NodeId를 대표 ID로 교체, 루프 요소(StartId==EndId) 제거.
Tolerance
NodeMergeTol = 1.0 mm
SpatialCellSizeMm = 200.0 mm
진단 코드
NODE_MERGED
HiTess 참조
Pipeline/NodeInspector/InspectEquivalenceNodes.cs
Pipeline/Utils/SpatialHash.cs, UnionFind.cs

Stage 4 IntersectionStage — 교차점 분할
목적
서로 교차하는 두 빔에 교차점 노드를 삽입해 구조적 연결을 표현합니다.
알고리즘 — Dan Sunday 3D 최근접점
1) ElementSpatialHash로 서로 가까운 요소 쌍 후보만 추출.
2) Dan Sunday segment-segment 알고리즘으로 두 선분 사이 최소 거리 계산.
3) 거리 < IntersectionTol(0.1mm) → 교차로 판정 → 교차점에 새 노드 삽입, 두 빔 각각 2분할.
4) 수렴 루프: 새 노드 삽입 후 또 다른 교차가 생길 수 있으므로 반복 (최대 50회).
Tolerance

Stage 6 GroupConnectStage — 그룹 연결
목적
같은 구조물이지만 좌표가 이어지지 않아 분리된 요소 그룹들을 연결합니다.
알고리즘
1) UnionFind로 BeamElement 연결 그룹 식별.
2) 가장 큰 그룹을 메인 그룹으로 선정.
3) 비메인 그룹의 각 요소에서 메인 그룹 최근접 요소 탐색.
4) 거리 < UboltSnapMaxDist×2(100mm) 이면 중간 연결 노드 생성 (NodeTags.Boundary).

Stage 7 UboltRbeStage — U-bolt RBE2 생성
목적
U-bolt가 구조 빔과 파이프를 묶는 강체 연결(RBE2)을 모델에 추가합니다.
알고리즘 — 2-phase snap
Phase 1 (힌트 기반): U-bolt 힌트 좌표 근처 Structure 빔 위 점으로 snap.
Phase 2 (fallback): Pipe 끝점 → 가장 가까운 Structure 노드 탐색, 거리 < UboltSnapMaxDist(50mm) → snap.
Snap 성공: RigidElement(Remark='UBOLT') 생성.
Snap 실패: NodeTags.Boundary 태그 + 경고 진단.
Tolerance

의존성 방향 — 왜 단방향인가?
설계 이유

순환 참조를 금지하는 이유: 변경의 영향 범위를 예측 가능하게 하기 위해서입니다. Core를 수정하면 모든 프로젝트가 영향받지만, Cli를 수정해도 Core는 영향받지 않습니다.

각 프로젝트 역할:

  Cmb.Core — 순수 도메인. 외부 NuGet 전혀 없음. 가장 안정적.

  Cmb.Io — 입출력. CSV 읽기, BDF/JSON 쓰기.

  Cmb.Pipeline — 알고리즘. 8개 파이프라인 스테이지, 공간 자료구조.

  Cmb.Cli — 진입점. 사용자 인터페이스(커맨드라인).
